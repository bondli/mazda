# mazda — Spec 阶段：需求澄清

你正在执行 mazda 工作流的 **第一阶段：Spec（需求澄清）**。

## 你的任务

1. **读取当前工作流状态**
   - 读取 `.mazda/state.json`，检查 `phase` 字段
   - 如果 phase 不是 `idle`，提示用户当前有未完成的需求，询问是否要放弃并重新开始

2. **获取需求名**
   - 询问用户为本次需求起一个简短的英文名称（用于目录命名，如 `user-login`、`dark-mode`）
   - 需求名只允许字母、数字和连字符

3. **澄清需求**
   - 请用户描述需求，可以是一句话或详细说明
   - 基于用户描述，主动追问以下关键点（不要一次全问，自然对话）：
     - 背景：为什么要做这件事？
     - 目标用户：谁会用到这个功能？
     - 范围：具体做哪些？明确不做哪些？
     - 验收标准：怎么判断做完了、做对了？

4. **生成 spec.md**
   - 澄清充分后，生成结构化的需求文档，格式如下：

   ```markdown
   # Spec: <需求名>

   ## 背景
   <为什么要做>

   ## 目标
   <期望达到的结果>

   ## 范围
   ### 在范围内
   - <具体要做的事项>

   ### 不在范围内
   - <明确不做的事项>

   ## 验收标准
   - [ ] <可验证的完成条件>

   ## 备注
   <其他补充信息>
   ```

   - 将文档写入 `.mazda/<需求名>/spec.md`
   - 更新 `.mazda/state.json`：phase 设为 `spec`，requirement 设为需求名

5. **等待确认**
   - 展示 spec.md 内容，告知用户：
     > "请审阅以上 Spec 文档。如果没有问题，回复"确认"或"approve"即可进入 Plan 阶段；如需修改，直接告诉我。"
   - 识别用户的确认意图（"approve"、"确认"、"没问题"、"ok"、"继续"、"go ahead"、"执行下一步"等）
   - 支持带条件确认（如"把验收标准改成xxx，然后确认"），先修改再确认
   - 确认后：更新 `.mazda/state.json` 的 `approved` 数组加入 `"spec"`，记录 `spec.approved_at`，然后自动开始 Plan 阶段（调用 mazda plan 的逻辑）

## 注意事项
- 始终保持对话自然，不要用表单式追问
- spec.md 是后续所有阶段的输入基准，要确保准确完整
- 如用户已提供详细需求，可跳过部分追问，直接生成草稿让用户确认
