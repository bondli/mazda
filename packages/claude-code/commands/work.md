# mazda — Work 阶段：执行开发

你正在执行 mazda 工作流的 **第三阶段：Work（执行开发）**。

## 前置检查

1. 读取 `.mazda/state.json`
2. 确认 `approved` 数组包含 `"spec"` 和 `"plan"`，否则提示用户先完成前置阶段
3. 读取 `.mazda/<需求名>/plan.md` 作为唯一执行依据

## 你的任务

### 执行原则
- **严格遵守 plan 范围**：只做 plan.md 任务列表中批准的事
- **逐任务执行**：每完成一个 Task，在对话中汇报进度
- **遇到范围外问题时**：先暂停，在对话中描述问题和建议方案，等用户确认后再处理
- **遇到 plan 中的歧义时**：先暂停询问，不要自行假设

### 执行流程

1. 宣布开始，列出将要执行的任务列表
2. 按顺序逐 Task 执行：
   - 宣布"开始 Task N：<任务名>"
   - 执行代码修改
   - 汇报"Task N 完成：<简述做了什么，涉及哪些文件>"
3. 所有 Task 完成后，生成执行日志

### 生成 work-log.md

```markdown
# Work Log: <需求名>

## 执行摘要
所有任务已完成，共修改 X 个文件。

## 任务执行记录

### Task 1: <任务名>
**状态**：✅ 完成
**实际变更**：
- 新增 `src/xxx.ts`：<做了什么>
- 修改 `src/yyy.ts`：<改了什么>

### Task 2: <任务名>
**状态**：✅ 完成 / ⚠️ 有偏差
**实际变更**：...
**偏差说明**：<如果实际执行与 plan 有差异，说明原因>

## 范围外处理记录
<如有用户临时批准的额外修改，在此记录>

## 变更文件汇总
- `src/xxx.ts`（新增）
- `src/yyy.ts`（修改）
```

写入 `.mazda/<需求名>/work-log.md`，更新 state.json phase 为 `work`。

4. **通知进入 Review**
   > "开发已完成，正在进入 Review 阶段..."
   - 自动触发 Review 阶段逻辑

## 注意事项
- 不允许在没有用户确认的情况下做 plan 以外的修改
- 如果某个 Task 无法完成（依赖缺失、技术阻碍等），立即报告，不要跳过或静默处理
