# mazda — Plan 阶段：制定执行计划

你正在执行 mazda 工作流的 **第二阶段：Plan（制定执行计划）**。

## 前置检查

1. 读取 `.mazda/state.json`
2. 确认 `approved` 数组包含 `"spec"`，否则提示用户先完成 Spec 阶段
3. 读取 `.mazda/<需求名>/spec.md` 作为输入

## 你的任务

1. **分析现有代码**
   - 扫描项目结构，理解当前实现
   - 识别与本次需求相关的文件和模块

2. **制定任务列表**
   - 将需求拆解为具体、可执行的任务
   - 每个任务包含：
     - 任务编号和名称
     - 具体操作描述
     - 涉及的文件（新增/修改/删除）
     - 预期产出
   - 标注任务间的依赖关系和建议执行顺序

3. **评估影响范围**
   - 列出所有会被改动的文件
   - 说明改动类型：新增 / 修改 / 删除

4. **识别风险**
   - 潜在的兼容性问题
   - 需要特别注意的边界情况
   - 依赖的外部因素

5. **生成 plan.md**，格式如下：

   ```markdown
   # Plan: <需求名>

   ## 基于
   Spec: `.mazda/<需求名>/spec.md`

   ## 任务列表

   ### Task 1: <任务名>
   **操作**：<具体做什么>
   **文件**：
   - 新增：`src/xxx.ts`
   - 修改：`src/yyy.ts`
   **产出**：<完成后能看到什么>

   ### Task 2: <任务名>
   ...

   ## 文件影响汇总
   | 文件 | 操作 | 涉及任务 |
   |------|------|----------|
   | `src/xxx.ts` | 新增 | Task 1 |

   ## 风险说明
   - <风险1>：<说明和应对>

   ## 执行顺序
   Task 1 → Task 2 → Task 3（Task 3 依赖 Task 2 完成）
   ```

   - 写入 `.mazda/<需求名>/plan.md`
   - 更新 `.mazda/state.json`：phase 设为 `plan`

6. **等待确认**
   - 展示 plan.md，告知用户：
     > "请审阅以上执行计划。确认后我将严格按计划开发，计划之外的修改需要你额外确认。回复"确认"或"approve"进入 Work 阶段。"
   - 识别确认意图（含带条件确认），确认后更新 state.json 并自动开始 Work 阶段

## 注意事项
- plan.md 是 Work 阶段的唯一执行依据，必须足够具体
- 任务粒度要适中：太粗无法追踪，太细增加维护成本
- 如有不确定的实现方案，在风险说明中列出并请用户决策
